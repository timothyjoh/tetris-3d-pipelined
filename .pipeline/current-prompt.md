# Update Project Status

You are the Status Reporter. Your job is to update the project's STATUS.md at the root — a living summary of everything the pipeline has built so far.

## Context — Read These First

1. **Existing STATUS.md** (if it exists) — preserve all previous phase sections
2. **Phase Spec**: `docs/phases/phase-3/SPEC.md` — what this phase set out to do
3. **Phase Plan**: `docs/phases/phase-3/PLAN.md` — how it was implemented
4. **Phase Review**: `docs/phases/phase-3/REVIEW.md` — code review findings
5. **Phase Reflections**: `docs/phases/phase-3/REFLECTIONS.md` — lessons and forward look
6. **Test output**: Run the project's test suite (check AGENTS.md for the test command) and capture results
7. **Project Brief**: `BRIEF.md` — for overall progress context

Current phase: 3

## Output

Write to `STATUS.md` in the **project root** (NOT in docs/phases/).

## Format

The file should have a header section and then a section per completed phase:

```markdown
# Project Status

> Auto-generated by cc-pipeline. Last updated after Phase N.

## Overview
[One-liner from BRIEF.md about what this project is]

**Current state:** [What a user can do RIGHT NOW with this project]

---

## Phase [N]: [Phase Name]

### What We Built
- [Concrete features/functionality delivered]
- [User-visible changes]
- [New endpoints, pages, components, etc.]

### How to Run It
- [Install/setup commands if changed]
- [How to start the app]
- [What to expect when you run it — what you'll see]

### Review Findings
- [Key issues found during code review]
- [What was fixed vs. deferred]
- [Any MUST-FIX items that were addressed]

### What's Next
- [Improvements planned for future phases]
- [Known limitations to address]
- [Features coming up per the brief]

### Test Coverage
- [Test command and results summary (X passing, Y failing)]
- [Coverage percentage if available]
- [Key E2E/integration tests added this phase]
- [Areas not yet covered]

---
```

## Rules

1. **Append, don't replace.** Keep all previous phase sections intact. Only add the new phase section.
2. **Update the Overview** section's "Current state" to reflect the latest capabilities.
3. **Be concrete.** "Added user auth" not "Implemented authentication subsystem." Write for a human checking in on progress.
4. **Actually run the tests.** Don't guess at coverage — execute the test command and report real numbers.
5. **Be honest about problems.** If the review found issues or tests are failing, say so. This is a status report, not a press release.
